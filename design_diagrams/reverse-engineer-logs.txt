## 1. BUwise Operations — `IFP_EAMOrg_OraBU_Default_Lookup`

**Exists in:** Both `IFP_PO_INBOUN_INTEGR_ATP` and `IFP_PO_RECEI_INBOU_INTEG_ATP`  
**Artifact:** DVM lookup table IFP_EAMOrg_OraBU_Default_Lookup.dvm  
**Purpose:** Maps an EAM Organization Name → Oracle Business Unit and all associated Oracle ERP parameters.

### DVM Schema (13 columns)

| Column | Description |
|---|---|
| `EAMOrgName` | **Lookup key** — EAM Organization short code (e.g., `CDHO`, `ACORN`) |
| `ORACompanyCode` | Oracle Company Code (e.g., `CHO`, `CAC`) |
| `ORACompanyName` | Human-readable company name (e.g., `Canadian Head Office`) |
| `ORABusinessUnit` | **Target BU** — either `CA BU` or `US BU` |
| `ORALegalEntityCode` | Legal entity code (e.g., `ICO`) |
| `ORALegalEntityName` | Legal entity name (e.g., `Interfor Corporation`) |
| `DocumentTypeCode` | Oracle PO document type (e.g., `STANDARD`) |
| `Buyer` | Buyer role/user (e.g., `Adapter Prod, ERP`) |
| `InitiatingParty` | Initiating party in PO header (always `BUYER`) |
| `ShiptoOrganization` | Oracle inventory org code (`CABUINV` or `USBUINV`) |
| `ShipToLocation` | Physical ship-to location |
| `DeliverToLocation` | Deliver-to location |
| `BillToLocation` | Bill-to location |

### BU Values in Use

| BU | ShiptoOrg | Represents |
|---|---|---|
| `CA BU` | `CABUINV` | Canadian operations |
| `US BU` | `USBUINV` | US operations |

**Examples from the data:**
- `CDHO` → `CA BU`, company `CHO` (Canadian Head Office), ship-to: `Interfor - Metrotown Head Office`
- `ACORN` → `CA BU`, company `CAC` (Acorn), ship-to: `Interfor - Acorn Division`
- Various other orgs → `US BU` / `USBUINV`

### How It Is Used
During PO/receipt data transformation (XSL mappers in validation and import integrations), `dvm:lookupValue()` is called with the EAM Org Name to resolve all Oracle ERP fields — BU, legal entity, document type, buyer, ship-to organization — needed to create the Oracle ERP PO or Receipt. **This is the core of BU-wise routing**: the integration does not hardcode BU logic; it drives everything from this single lookup table.

> ✅ **Accuracy:** Confirmed from IFP_EAMOrg_OraBU_Default_Lookup.dvm in both projects (identical schema, same data). Both projects share the same lookup revision 0.

## 2. Batchwise Processing — `IFP_EAM_PO_BATCH_PROCE_INTEG` (PO Inbound)

**Project:** `IFP_PO_INBOUN_INTEGR_ATP`  
**Integration ID:** `IFP_EAM_PO_BATCH_PROCE_INTEG|01.00.0000`  
**Pattern:** App-Driven Orchestration, MEP08 (subscribe to event)  
**Tracking field:** `parameter1`

### Trigger
Subscribes to the `TriggerNewPObatch` application event (`TRIGGERNEWPOBATCH|1`).

### Flow Logic
1. **Event received** — `TriggerNewPObatch` fires with a batch ID.
2. **ATP stored proc `ponewbatchfetch`** — calls the procedure (from DB package `ADMIN.XXIFP_POINBOUND_INT_PKG`) to fetch the batch record. Returns a table type `P_PO_BATCH_TABTYPE`, each item (`P_PO_BATCH_TABTYPE_ITEM`) containing:
   - `BATCH_ID` (NUMBER)
   - `BATCH_STATUS` (VARCHAR2)
   - `BATCH_FILE_NAME` (VARCHAR2)
   - `CREATION_DATE`, `CREATED_BY`, `LAST_UPDATED_DATE`, `LAST_UPDATE_BY`
3. **For-each loop** — iterates over each `P_PO_BATCH_TABTYPE_ITEM`.
4. **Mapper** — Maps `BATCH_ID` → `pbatchId` and sets batch status to `DRAFT` via ATP proc `updatebatchstatusasDraft`.
5. **Calls Stage Data Loading** (`InvokePOStageloadintegration`) — passes the batch record downstream for PO line staging.

> ✅ **Accuracy:** Confirmed from info.json (subscribes to `TRIGGERNEWPOBATCH|1|sub`), `ponewbatchfetch` WSDL (showing `ADMIN.XXIFP_POINBOUND_INT_PKG.L_PO_BATCH_RECTYPE`), and mapper XSL files.

**Companion:** `IFP_EAM_PO_BATCH_REPRO_INTEG` — "IFP EAM PO Batchwise Reprocessing Integration" — the reprocess variant following the same pattern.

## 3. Batchwise Processing — `IFP_EAM_PO_RECE_BATC_PROC_INTE` (PO Receipt)

**Project:** `IFP_PO_RECEI_INBOU_INTEG_ATP`  
**Integration ID:** `IFP_EAM_PO_RECE_BATC_PROC_INTE|01.00.0000`  
**Pattern:** App-Driven Orchestration, MEP08 (subscribe to event)  
**Tracking field:** `p_new_batch_id`

### Trigger
Subscribes to the `callprioritywisePORCPT` application event (`CALLPRIORITYWISEPORCPT|1`). This makes it the **consumer** of what the Prioritywise integration publishes (see below).

### Flow Logic
1. **Event received** — `CALLPRIORITYWISEPORCPT` fires with payload: `p_new_batch_id`, `p_new_import_id`, `p_new_priority_col`.
2. **Mapper 1** — Extracts `p_new_batch_id` from the event and maps to `pbatchId` for the Stage Data Loading call, and to `P_BATCH_ID` for the ATP status update.
3. **ATP `BatchstatusDRAFTupd`** — Updates receipt batch status to `DRAFT` with `P_BATCH_ID` and `P_BATCH_STATUS`.
4. **Calls Stage Data Loading** — calls `callPORCPTstagedataloading` REST endpoint `/ponewbatchsplit` to stage the receipt batch for processing.

> ✅ **Accuracy:** Confirmed from info.json (subscribes to `CALLPRIORITYWISEPORCPT|1|sub`), batchidexpr.properties (XPath `/ns13:execute/ns10:request-wrapper/ns10:p_new_batch_id`), XSL mappers showing `P_BATCH_ID` and `P_BATCH_STATUS`, and `.jca` file showing `/ponewbatchsplit`.

### Branch Details

**Branch A — `pbatchid > 0` (batch already known):**
- XSL condition: `/ns13:execute/ns13:QueryParameters/ns16:pbatchid > 0`
- Transformer maps: `pbatchid` → `p_new_batch_id` in `CALLPRIORITYWISEPORCPT1` event
- Publishes `CALLPRIORITYWISEPORCPT1` → consumed by `IFP_EAM_PO_RECE_BATC_PROC_INTE`

**Branch B — priority-based (no known batch ID, use priority to find next batch):**
- ATP Stored Proc **`PORCPTpriority`** called with input `P_ARG_PRIORITY = priority` query param
- Returns: `P_NEW_BATCH_ID`, `P_NEW_IMPORT_ID`, `P_NEW_PRIORITY_COL` (the next batch to process at that priority level)
- Transformer maps these outputs → `CALLPRIORITYWISEPORCPT2` event payload
- Publishes `CALLPRIORITYWISEPORCPT2` → consumed by `IFP_EAM_PO_RECE_BATC_PROC_INTE`

## 5. Event Type: `CALLPRIORITYWISEPORCPT` — The Integration Bus

This application event is the **asynchronous coupling** between Prioritywise Reprocess and Batchwise Processing in the Receipt project:

```
IFP_EAM_PRIORITY_REPROCES          IFP_EAM_PO_RECE_BATC_PROC_INTE
        │  PUBLISHER                              SUBSCRIBER │
        │                                                     │
        ├──► CALLPRIORITYWISEPORCPT1 ──────────────────────►│
        └──► CALLPRIORITYWISEPORCPT2 ──────────────────────►│
```

**Event payload schema:**
- `p_new_batch_id` — the new receipt batch ID to process
- `p_new_import_id` — associated import ID
- `p_new_priority_col` — priority column value used for ordering

### 2. What Does "Split" Mean?

The "split" is performed by the Oracle ATP stored procedure `XXIFP_PORCPT_SPLIT_INSERT_DATA`, inside the database package `ADMIN.XXIFP_PORCPT_INB_INT_PKG`.

**Conceptually:** A receipt batch arriving from EAM Hexagon is stored initially as a single batch header record in ATP. This batch can contain **multiple PO receipt lines**. The split procedure reads that batch record and **decomposes it into individual staging rows** — one per receipt line — inserting them into the receipt staging table. This makes each line independently addressable for the validation and import phases that follow.

The orchestration block inside the integration is explicitly named `callbatchsplitting` (from project.xml), confirming this is the design intent.

> ✅ **Accuracy:** Confirmed from mainstageload_REQUEST.jca (`ProcedureName=XXIFP_PORCPT_SPLIT_INSERT_DATA`) and the try-block name `callbatchsplitting` in project.xml.

## Diagram 1 — PO Inbound: Main Pipeline (Technical Description)

**This diagram covers 4 of the 11 endpoints: `/ponewbatchsplit` (PO instance), `/purchasevalidation`, `/poinboundmain` (PO Import instance), `/poinboundmain` (EAM Status Update instance).**

---

### Step 1 — PO Data Extraction (no endpoint, schedule-triggered)

**Integration:** `IFP_EAM_PO_DATA_EXTRA_INTEG` · MEP07 Scheduled

The pipeline begins with a scheduled trigger. The integration runs on a configured schedule, calls EAM Hexagon via `IFP_PO_EAM_REST_CONNEC` to fetch PO header and line grid data from the Hexagon `userdefinedscreenservices` REST API. Lookup tables (`IFP_EAMOrg_OraBU_Default_Lookup`, `IFP_EAMTo_OracleUOM_Mapping_Lookup`) map EAM organization codes to Oracle Business Unit parameters. Fetched records are written to the ATP database (`IFP_PO_ATP_DB_CONNEC`, package `ADMIN.XXIFP_POINBOUND_INT_PKG`). After staging, it **publishes the `TRIGGERNEWPOBATCH` application event**, delivering the new batch ID to all subscribers. No REST endpoint is exposed here.

> ✅ Confirmed: info.json `mepType=MEP07`, `eventtypes: TRIGGERNEWPOBATCH|1|pub`, connections include `IFP_PO_EAM_REST_CONNEC`, `IFP_PO_ATP_DB_CONNEC`.

---

### Step 2 — Batchwise Processing receives event, calls `/ponewbatchsplit`

**Integration:** `IFP_EAM_PO_BATCH_PROCE_INTEG` · MEP08 Event Subscriber

Subscribes to `TRIGGERNEWPOBATCH`. Upon receiving the event, it calls the ATP stored procedure `ponewbatchfetch` (package `ADMIN.XXIFP_POINBOUND_INT_PKG`) to retrieve the batch record — which contains `BATCH_ID`, `BATCH_STATUS`, `BATCH_FILE_NAME`, `CREATION_DATE`, `CREATED_BY`, `LAST_UPDATED_DATE`, `LAST_UPDATE_BY`. It then updates the batch status to DRAFT in ATP.

Then it fires **Endpoint ①: `POST /ponewbatchsplit`** via a **collocated ICS call** using `PRESEEDED_COLLOCATED_CONN_1741`:

| Property | Value |
|---|---|
| URI | `/ponewbatchsplit` |
| Target integration | `IFP_EAM_PO_STAG_DATA_LOAD_INTE\|01.00.0000` |
| HTTP Method | POST |
| Query parameter sent | `pbatchId` = `p_new_batch_id` from event payload (XSL confirmed) |
| Invocation style | NOTIFICATION (fire-and-forget — Batchwise Processing does not wait) |
| Auth | None (intra-OIC collocated, security bypass) |
| Context | `INTRA_PROJECT` |

> ✅ Confirmed: InvokePOStageloadintegration_REQUEST.jca — `integrationContext=INTRA_PROJECT`, `integration_id=IFP_EAM_PO_STAG_DATA_LOAD_INTE|01.00.0000`, `ResourceURI=/ponewbatchsplit`, `HttpVerb=POST`.

### Step 3 — Stage Data Load receives `/ponewbatchsplit`, splits batch

**Integration:** `IFP_EAM_PO_STAG_DATA_LOAD_INTE` · MEP00 · App-Driven Orchestration

This integration **exposes** `/ponewbatchsplit` as its REST trigger via `IFP_PO_OIC_REST_CONNEC` (Basic Auth, adapter=REST). The trigger is activated as `NOTIFICATION` (invocationStyle), so it accepts the call and processes asynchronously.

**Internal orchestration sequence** (try block `callbatchsplitting`):

1. **Transformer** maps `pbatchId` → `P_BATCH_ID` for the ATP split call
2. **ATP Call `mainstageload`** → `ADMIN.XXIFP_POINBOUND_INT_PKG.XXIFP_POSPLIT_INSERT_DATA` (equivalent split proc for PO)
   - IN: `P_BATCH_ID` (NUMBER)
   - OUT: `P_BATCH_ID`, `P_OUT_STATUS`, `P_ERROR_MSG`
3. **Transformer** maps `pbatchId` → `P_BATCH_ID` + hardcoded `P_BATCH_STATUS = "READY"`
4. **ATP Call `updateReadybatchstatus`** → batch status update proc
   - IN: `P_BATCH_ID`, `P_BATCH_STATUS`
   - OUT: `P_OUT_STATUS`, `P_ERROR_MSG`
5. **Transformer** maps payload for the validation call
6. **Collocated ICS call** → fires **Endpoint ②: `POST /purchasevalidation`** (next step)

Error handling: inner catchAll = ehStop; outer catchAll = email notification to distribution list from `IFP_Common_Utility_Lookup` RICE_ID `INT-PO-01a`.

### Step 4 — Validation receives `/purchasevalidation` (synchronous)

**Integration:** `IFP_EAM_PO_VALIDA_INTEGR` · MEP00

Exposes **Endpoint ②: `POST /purchasevalidation`** via `IFP_PO_OIC_REST_CONNEC` (Basic Auth).

**Critical characteristic: `invocationStyle=REQUEST_RESPONSE`** — this is the **only synchronous custom endpoint in the PO Inbound pipeline**. The Stage Data Load integration BLOCKS and waits for the JSON response: `{ "pbatchid", "pprevoicintid", "Argument1", "Argument2", "Argument3" }`.

**Query parameters received:** `pbatchId`, `poicintid`

**Internal operations:**
- BIP SOAP calls (`IFP_PO_ORACL_BIP_SOAP_CONNE`): validates items, suppliers, COA/GL codes
- ATP reads: checks staging rows inserted by split proc
- Conditional item creation: calls `CreateMasterItems` (ERP REST, FBDI) and `CreateChildItemOrg` if items don't exist
- On pass: fires **Endpoint ③: `POST /poinboundmain`** to trigger Import
- On fail: updates ATP batch status, publishes error event

**Query parameters:** `pbatchId`

### Step 6 — Import Callback → Status Event → `/poinboundmain` (Instance B — EAM Status Update)

**Integration (callback):** `IFP_PO_IMPOR_CALLB_16_FEB` · MEP00 · ERP application event subscriber

After the ESS PO Import job completes, ERP fires an application event that this integration catches. It:

**Integration (event handler):** `IFP_EAM_STATUS_EVENT_INTEGR` · MEP08 · subscribes to `PUBLISH_EAMPOSTATUSUPDAT`

### `/callbackreprocess` — PO Import Batchwise Reprocess instance

**Integration:** `IFP_EAM_PO_BATCH_REPRO_INTEG` · MEP00 · subscribes to `TRIGGERNEWPOBATCH`

**Distinguishing feature:** Unlike the Receipt callback reprocess (which takes `loadrequestid + importrequestid`), this PO batchwise reprocess instance accepts **only `importid`** — a single import-request identifier that allows re-entry into the import callback phase without re-running validation.

**Flow:** When the `TRIGGERNEWPOBATCH` event fires (which it does for both the main Batchwise Processing path AND this reprocess subscriber), this integration targets batches that have already been validated but whose import job needs re-submission. > ✅ Confirmed: batchwisereprocess_REQUEST.jca — `ResourceURI=/callbackreprocess`, `QueryParameters=importid` (only), `invocationStyle=NOTIFICATION`, `integration_id=IFP_EAM_PO_BATCH_REPRO_INTEG`. info.json — `eventtypes: TRIGGERNEWPOBATCH|1|pub` (this integration also fires the same event when re-triggering, creating a loop back through Batchwise Processing).

> ✅ Confirmed: callbackreprocess_REQUEST.jca (PO Import Callback Reprocess) — `QueryParameters=loadrequestid, importrequestid`. Two separate integration instances, same URI, same connection base URL — differentiated by parameter cardinality.

- **HTTP verb:** POST, **style:** NOTIFICATION
- **Parameters:** `?pbatchid` + `?pimportid` + `?priority`
- **Connection serving it:** `IFP_PO_RECEI_OIC_REST_CONNE`

**Internal logic (CBR — Content-Based Routing):**

Both branches eventually trigger `IFP_EAM_PO_RECE_BATC_PROC_INTE` (the main Batchwise Processing subscriber), which then calls `/ponewbatchsplit` to re-enter the pipeline from Stage Data Load.
